# 编码规范指南

你是编码规范指南。你提供 Clean Architecture 的分层编码规范，帮助编码 subagent 按层正确实现代码。

---

## 分层总览

基于 Clean Architecture，从外到内分为 6 层：

| 层 | 名称 | 职责 | Spec 文件 |
|---|------|------|----------|
| 1 | 入口层 | 接收外部请求、参数校验、响应格式化 | `specs/entry.md` |
| 2 | 编排层 | 业务流程编排、事务边界管理 | `specs/orchestration.md` |
| 3 | 领域层 | 核心业务逻辑、领域模型、业务规则 | `specs/domain.md` |
| 4 | 数据层 | 数据存取、ORM/查询映射 | `specs/data.md` |
| 5 | 集成层 | 外部服务调用、超时、降级 | `specs/integration.md` |
| 6 | 基础设施层 | 配置、中间件、工具函数 | `specs/infrastructure.md` |

---

## 分层依赖规则

这是核心约束，所有编码必须遵守：

1. **外层可调用内层**：入口 → 编排 → 领域 → 数据（单向依赖）
2. **内层不可调用外层**：领域层不能调用入口层，数据层不能调用编排层
3. **集成层和基础设施层可被任意层调用**：它们是横切关注点
4. **跨层调用必须通过接口**：依赖倒置原则——内层定义接口，外层实现接口

```
依赖方向（箭头表示"可调用"）：

入口层 → 编排层 → 领域层 → 数据层
  │         │         │         │
  └─────────┴─────────┴─────────┘
              ↓        ↓
          集成层   基础设施层
```

---

## 编码 SOP

编码 subagent 按以下顺序工作：

```
1. 读 spec.md 确认需求范围
2. 确定本次变更涉及哪些层
3. 加载对应层的 Spec 文件（按需 read specs/{layer}.md）
4. 从内到外编码：先领域层 → 数据层 → 集成层 → 编排层 → 入口层
5. 每层编码时对照 Spec 检查合规性
```

从内到外的顺序确保每一层的依赖已经就绪，避免编码时发现下层接口不存在而回退。

## Spec 文件说明

`specs/` 目录下每个文件对应一层的详细编码规范，包含：

- **职责边界**：该层做什么、不做什么
- **依赖规则**：允许调用谁、被谁调用
- **禁止规则**：明确列出不允许的做法
- **编码规范**：命名、结构、错误处理等具体要求
- **正面示例 / 反面示例**：帮助 subagent 理解正确实践

编码 subagent 应按需读取涉及的层的 Spec 文件，不需要一次全部加载。

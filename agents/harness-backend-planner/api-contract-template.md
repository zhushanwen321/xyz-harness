## 3. 通用约定

### 3.1 请求头

| Header | 值 | 必填 | 说明 |
|--------|------|------|------|
| `Content-Type` | `application/json` | 是（有请求体时） | 请求体格式 |
| `Authorization` | `Bearer <token>` | 视端点而定 | 认证 Token |
| `Accept-Language` | `zh-CN` / `en-US` | 否 | 国际化语言偏好 |
| `X-Request-Id` | UUID v4 | 否 | 请求追踪 ID，建议前端生成 |

> **填写指引**：列出所有端点共用的请求头。如果有自定义请求头（如租户标识、客户端版本），也在此定义。

### 3.2 响应结构

**成功响应**：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

> **填写指引**：
> - 定义统一的成功响应外壳，`data` 字段承载实际业务数据。
> - `code` 为业务状态码，`0` 表示成功，非 `0` 表示失败（与 HTTP 状态码独立）。
> - `message` 为人类可读的提示文案，前端可用于展示。
> - 如果项目已有统一的响应格式，直接引用已有的格式定义，不要重新发明。

**失败响应**：

```json
{
  "code": 40001,
  "message": "参数校验失败",
  "details": [
    {
      "field": "email",
      "message": "邮箱格式不正确"
    }
  ]
}
```

> **填写指引**：
> - `details` 用于参数校验等需要逐字段反馈的场景，非校验类错误可省略。
> - `message` 应为面向用户的提示文案，而非技术错误信息（不要暴露堆栈、SQL 等）。

### 3.3 分页规范

**请求参数**：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `page` | integer | 否 | 1 | 页码，从 1 开始 |
| `page_size` | integer | 否 | 20 | 每页条数，最大 100 |

**响应结构**：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 20,
    "has_more": false
  }
}
```

> **填写指引**：
> - 如果项目使用游标分页（cursor-based）而非偏移分页，替换为 `cursor` / `limit` 模式，并说明游标的编码格式。
> - `has_more` 用于前端判断是否显示"加载更多"。
> - 分页参数名必须与项目现有规范一致，不要自创命名。

## 4. 错误码定义

### 4.1 错误码段划分

| 段 | 范围 | 说明 |
|----|------|------|
| 通用错误 | `10000 - 19999` | 参数校验、认证、权限等 |
| 业务错误 | `20000 - 29999` | 本需求的业务错误 |

> **填写指引**：
> - 错误码是纯数字，独立于 HTTP 状态码。
> - 如果项目已有错误码段划分，直接遵循，不要重新分段。
> - 每个需求的业务错误码在本段范围内从起始值顺序分配。

### 4.2 通用错误码

| 错误码 | HTTP 状态码 | 含义 | 前端处理建议 |
|--------|------------|------|-------------|
| 0 | 200 | 成功 | - |
| 10001 | 400 | 请求参数格式错误 | 展示 `message` 或 `details` |
| 10002 | 401 | 未认证（Token 缺失或无效） | 跳转登录页 |
| 10003 | 403 | 无权限访问该资源 | 展示无权限提示 |
| 10004 | 404 | 资源不存在 | 展示 404 页面 |
| 10005 | 409 | 资源冲突（如唯一约束） | 展示 `message` 提示用户 |
| 10006 | 429 | 请求频率超限 | 展示限流提示，延迟重试 |
| 10007 | 500 | 服务端内部错误 | 展示通用错误提示 |

> **填写指引**：列出项目通用的错误码。这些错误码跨需求复用，前端应有统一的拦截处理逻辑。

### 4.3 业务错误码

| 错误码 | HTTP 状态码 | 含义 | 触发条件 | 前端处理建议 |
|--------|------------|------|---------|-------------|
| 20001 | 400 | （示例）订单状态不允许此操作 | 订单已取消时尝试支付 | 提示用户刷新页面 |
| 20002 | 409 | （示例）库存不足 | SKU 库存 < 请求数量 | 引导用户调整数量 |

> **填写指引**：
> - 每个业务错误码必须说明触发条件，前端才能针对性地处理。
> - HTTP 状态码应与错误语义匹配：客户端错误用 4xx，服务端错误用 5xx。
> - 错误码一旦分配并交付前端使用，不应复用（即不应修改已有错误码的含义），新增即可。

## 5. 端点详细设计

> **填写指引**（本章节是合约的核心，必须精确到每个字段）：
>
> 1. **字段命名一致性**：所有字段名必须与"业务统一语言"一致。如果 plan-backend.md 或项目文档中定义了术语表，字段命名必须遵循。例如用户称呼为"客户"，字段名应为 `customer_id` 而非 `user_id`。
>
> 2. **类型精确**：每个字段必须标注 JSON 类型（`string` / `integer` / `number` / `boolean` / `array` / `object`），以及额外约束（枚举值、格式、范围）。避免模糊的类型标注如 "数字"——要明确是 `integer` 还是 `number`（允许小数）。
>
> 3. **必填/可选**：请求参数和响应字段都要标注是否必填。请求参数的"必填"意味着前端必须传值；响应字段的"必填"意味着后端保证一定返回该字段。
>
> 4. **示例值**：每个请求参数和响应字段都应给出示例值，示例值必须真实合理，不要用 `"string"` / `0` / `"xxx"` 这种无意义占位。示例值是前后端对齐理解的重要手段。
>
> 5. **业务规则**：描述该端点特有的业务约束和边界情况，例如：
>    - 并发请求时的行为（乐观锁？返回冲突错误？）
>    - 数据上限（单次最多创建多少条？）
>    - 副作用（调用此接口会触发什么？邮件通知？状态流转？）
>
> 6. **注意事项**：面向前端开发者的提示，例如：
>    - 该接口是否幂等，失败后能否安全重试
>    - 是否需要轮询或配合 WebSocket 使用
>    - 响应数据量级预估，前端是否需要虚拟滚动
>    - 移动端/弱网环境下的特殊处理
>
> 7. **不要写实现代码**：本合约只描述"接口是什么"，不描述"后端如何实现"或"前端如何调用"。

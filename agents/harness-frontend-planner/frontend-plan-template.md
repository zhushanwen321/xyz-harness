> **填写指引**：
> - 组件分层策略决定了代码组织方式。写清楚每层的**边界判定规则**——什么时候是页面组件，什么时候该抽成业务组件。
> - 组件清单覆盖**本次需求涉及的所有新增和修改组件**。
> - 核心组件详细设计只写**交互复杂或有设计决策**的组件。简单的展示型组件不需要详细设计，在清单中说明即可。
> - Props 接口只写关键签名，不写完整实现。重点是**接口设计为什么是这样**——哪些属性是必须的，哪些是可选的，为什么。
> - Events 的设计反映组件间通信方式。如果一个组件发出很多事件，可能意味着职责过重。
> - 内部状态要明确归属——这直接决定了状态管理方案的选择。

  - {边界 1}：{描述} → {处理方式}
  - {边界 2}：{描述} → {处理方式}

### 3.2 状态分类

| 状态数据 | 类型 | 归属 | 持久化 | 共享范围 | 设计说明 |
|---------|------|------|--------|---------|----------|

**状态类型定义**：
- **UI 状态**：展开/折叠、选中标签、输入草稿、排序方式等纯展示控制数据

- **背景**：{为什么需要选择状态管理方案，有哪些状态共享场景}
- **决策**：{选择了什么方案（本地 ref / Composable / Pinia Store / 其他）}
- **备选方案**：

- **取舍理由**：{为什么选这个方案，放弃了什么}

### 3.4 异步状态处理策略

| 场景 | Loading 表现 | 错误处理 | 重试策略 | 设计说明 |
|------|-------------|---------|---------|----------|
| {数据加载} | {骨架屏/Spinner/进度条} | {toast / 内联错误 / 错误页面} | {自动重试/手动重试/无} | {为什么选这种处理方式} |
| {表单提交} | {按钮禁用+Loading} | {字段级错误 / 表单级错误} | {无/手动重试} | {防重复提交策略} |

### 3.5 数据流图

```
用户操作 → Component → Composable/Store → API Client → 后端
                          ↓
UI 更新 ← Component ← Composable/Store ← 响应数据 ←┘
```

> **填写指引**：
> - 交互场景设计是前端设计文档最核心的章节。每个场景必须覆盖**四种 UI 状态**（加载中/成功/失败/空数据），遗漏任何一种都会导致用户体验缺陷。
> - 状态分类表是防止"状态混乱"的关键工具。前端 bug 的第一大来源就是状态归属不清晰——"这个数据归谁管"、"从哪获取"、"什么时候更新"。
> - 异步状态处理策略不是可选项。现代前端几乎所有交互都涉及异步操作，每种异步操作必须有明确的 loading/error/success 处理方案。
> - 如果需求不涉及复杂状态管理（如纯展示页面），3.3 和 3.5 写"N/A — 本需求为纯展示型页面，无跨组件状态共享"。
